Exercício Prático

Requisitos do Projeto

O cliente solicita o desenvolvimento de um aplicativo simples para gerenciamento de tarefas pessoais, voltado para pessoas comuns que precisam controlar suas atividades diárias de maneira prática. Ele reforça que o aplicativo deve ser extremamente fácil de usar, visualmente limpo, minimalista e acessível até para quem tem pouca experiência com tecnologia.

Requisitos Funcionais

  • Adicionar novas tarefas com título e descrição opcional.
  • Marcar tarefas como concluídas.
  • Remover tarefas da lista.
  • Editar tarefas existentes.
  • Exibir separadamente tarefas pendentes e concluídas.
  • Filtrar tarefas por status.
  • Ordenar tarefas por prioridade ou data.
  • Manter tarefas salvas após fechar o app (persistência simulada).
  • Interface clara, intuitiva e agradável.

Requisitos Não Funcionais

  • Interface responsiva.
  • Navegação simples e rápida.
  • Design minimalista.
  • Desempenho suave.

Instruções da Atividade

Você deve simular como seria participar de um projeto real utilizando Scrum, mesmo sem construir nenhum software. Simule o processo completo: transforme os requisitos em user stories, organize um Product Backlog completo e planeje todas as sprints como se uma equipe estivesse realmente desenvolvendo o aplicativo.

1. Criar o Product Backlog

Você deve criar um backlog completo contendo:

  • Blocos de trabalho maiores, agrupando funcionalidades semelhantes.
  • User stories claras, simples e centradas no usuário.

Certifique-se de que o backlog represente tudo o que seria desenvolvido ao longo do projeto.

2. Planejar todas as sprints

Simule quantas sprints seriam necessárias e distribua as user stories de forma coerente. Para cada história, estime o esforço, justifique a escolha e defina prioridades. Garanta que cada sprint tenha uma meta clara que mostre a evolução do produto.

3. Criar o Sprint Backlog

Para cada sprint planejada, transforme as user stories selecionadas em tarefas menores. Registre dependências, possíveis impedimentos e a ordem lógica de execução, organizando o trabalho como se a equipe estivesse prestes a iniciar o desenvolvimento.

4. Cerimônias que não podem ser simuladas

  • Sprint Review: registre que ela existiria, mas não pode ser simulada porque não haverá incremento real.
  • Sprint Retrospective: registre que também não pode ser realizada, pois não houve execução real.

É recomendado revizar as Cerimônias acima afim de fixar o conteúdo, já que elas não podem ser simuladas.

5. Organização em Ferramentas

Use, se quiser, ferramentas como Jira, Trello ou YouTrack para visualizar seu backlog, distribuir as histórias e planejar as sprints, simulando o fluxo real de trabalho de uma equipe Scrum.
